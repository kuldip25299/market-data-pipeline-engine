# Glossary

Every technical term used in this repository, explained in plain
English with a real-world analogy. If you hit a term anywhere in the
docs or code comments that isn't here, please open an issue.

---

**Exchange**
A marketplace where buyers and sellers trade financial instruments
(stocks, futures, crypto). Analogy: a farmers' market, but for prices
instead of vegetables, with the exchange as the market organizer
matching buyers to sellers.

**Broker**
A company that connects individual traders/firms to the exchange, and
often relays exchange data onward. Analogy: a real-estate agent
standing between you and the property market.

**Tick**
A single market data event: one price/quantity update for one
instrument at one moment in time. Analogy: a single "ding" of a
scoreboard updating.

**Latency**
The time delay between something happening and a system reacting to
it. Not a single number -- usually described as a distribution (e.g.
"median latency 200 microseconds, 99th percentile 2 milliseconds").
Analogy: the delay between a starting gun firing and a sprinter
actually leaving the blocks.

**Throughput**
How much work a system completes per unit of time (e.g. "ticks per
second"). Analogy: how many cars a toll booth can process per minute,
regardless of how long any single car takes.

**Garbage Collection (GC)**
The process by which a language runtime automatically frees memory
that's no longer being used. Analogy: a cleaning crew that goes around
and removes items nobody is using anymore, so the room doesn't fill up.

**Memory Allocation**
Reserving a chunk of computer memory for a new piece of data.
Analogy: finding and reserving an empty locker for a new gym member.

**CPU Cache**
A small, extremely fast memory area inside the CPU that stores
recently/frequently used data so the CPU doesn't have to fetch it from
much slower main memory every time. Analogy: keeping the ingredients
you use most on your kitchen counter instead of walking to the pantry
each time.

**Cache Locality**
How likely related data is to already be sitting in the CPU cache when
needed, based on how it's laid out in memory. Analogy: keeping related
tools together in one drawer instead of scattered across the garage.

**Context Switching**
When a CPU pauses one task to run another, saving/restoring state each
time. Frequent switching adds overhead. Analogy: constantly being
interrupted while cooking one dish to briefly work on another -- you
lose time re-orienting yourself each time you switch back.

**Object Pool**
A collection of pre-allocated, reusable objects, handed out and
returned instead of being created and destroyed repeatedly. Analogy: a
library lending books instead of every reader buying and then
discarding a new copy.

**Memory Fragmentation**
When free memory becomes split into many small, non-contiguous
chunks, making it harder to allocate large blocks even though total
free memory is sufficient. Analogy: a parking lot with scattered empty
spaces too small individually to fit a bus, even though their combined
area would be enough.

**Producer**
A component that generates or emits data for others to use. In this
repository: whatever creates `Tick` objects.

**Consumer**
A component that receives and acts on data generated by a producer. In
this repository: whatever calls `process_tick`.

**Queue**
An ordered, first-in-first-out data structure used to hand data from
one part of a system to another. Analogy: a single-file line at a
coffee shop counter.

**Backpressure**
A signal (explicit or implicit, like a growing queue) that a consumer
can't keep up with a producer, used to slow the producer down or alert
operators. Analogy: a "wait to be seated" sign at a restaurant, telling
new customers to pause instead of overwhelming the kitchen.

**Zero Copy**
Techniques that avoid duplicating data in memory when moving it
between components, instead sharing a reference to the same underlying
bytes. Analogy: sharing one physical document by passing it around,
instead of photocopying it for every reader.

**Lock-Free**
Concurrent code designed to make progress without threads blocking
each other via locks, usually using specialized atomic operations
instead. Analogy: a self-service buffet where people serve themselves
without needing to take turns via a locked door.

**Shared Memory**
A region of memory that multiple separate processes can read/write
directly, avoiding the cost of copying data between them. Analogy: a
shared whiteboard in an office that multiple people can write on,
instead of each person keeping (and having to sync) their own private
notebook.

**Ring Buffer**
A fixed-size buffer that wraps around to the beginning once full,
reusing slots instead of growing indefinitely. Analogy: a circular
conveyor belt at a sushi restaurant -- plates keep cycling through a
fixed set of positions.

**SIMD (Single Instruction, Multiple Data)**
A CPU feature that applies the same operation to multiple pieces of
data simultaneously. Analogy: stamping ten envelopes at once with one
press of a multi-stamp tool, instead of stamping each individually.

**NUMA (Non-Uniform Memory Access)**
A multi-CPU architecture where each CPU has faster access to its
"local" memory and slower access to memory attached to other CPUs.
Analogy: it's faster to grab a snack from your own desk drawer than to
walk to a coworker's desk across the office.

**Kernel**
The core part of an operating system that manages hardware resources
(CPU, memory, devices) on behalf of all running programs. Analogy: the
building manager who allocates rooms and utilities to every tenant.

**Thread**
An independent sequence of execution within a program, potentially
running in parallel with other threads. Analogy: one worker among
several working simultaneously on parts of the same project.

**Coroutine**
A function that can pause its execution (at `await` points) and let
other code run, then later resume where it left off. Analogy: a chef
who puts a pot on to simmer, works on a different dish while waiting,
then comes back to the first pot when it's ready.

**AsyncIO**
Python's standard library framework for writing concurrent code using
coroutines, ideal for I/O-bound workloads (waiting on network, disk,
etc.) rather than CPU-bound ones. Used throughout this repository.

**WebSocket**
A network protocol that keeps a connection open for continuous,
two-way message exchange -- commonly used for real-time market data
feeds. Analogy: an open phone line you keep connected, instead of
hanging up and re-dialing for every new sentence.
