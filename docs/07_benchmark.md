# Chapter 7 — Benchmarking Naive vs Optimized

## Business Motivation

A claim like "the object pool makes this faster" is worthless to an
engineering organization without numbers attached. This chapter
defines exactly how we measure the two pipelines so that the
comparison is fair, reproducible, and honest about its limitations.

## Problem

We need to answer, with evidence:

1. Is the optimized pipeline actually faster?
2. How much does object allocation drop?
3. Does memory usage improve?
4. Are there any downsides (e.g. does pooling ever get *slower* for
   small workloads, due to pre-allocation overhead)?

## Naive Approach to Benchmarking (and why it's insufficient)

Running each pipeline once and comparing wall-clock time sounds
reasonable, but a single run is vulnerable to noise: background OS
processes, CPU frequency scaling, and Python's own JIT-adjacent
warm-up behaviors can all skew a single measurement.

## Production Approach: Controlled, Repeatable Comparison

`benchmarks/compare.py` runs both pipelines against **identical
workloads** (same tick count, same simulated symbol universe, same
random seed) back to back, and reports:

- Ticks Processed
- Objects Created / Objects Reused
- Peak Queue Size
- Elapsed Time
- Throughput (ticks/sec)
- Peak Memory (MB)

```bash
python benchmarks/compare.py 1000000 1000
```

## Reading the Results

Below is a real comparison table, generated by running
`python benchmarks/compare.py 1000000 1000` on the container used to
build this repository (a shared, virtualized CPU -- not tuned
hardware; see the note on hardware variance below).

| Metric               | Naive        | Optimized    |
|-----------------------|-------------:|-------------:|
| Ticks Processed        | 1,000,000    | 1,000,000    |
| Objects Created         | 1,000,000    | 1,001        |
| Objects Reused          | 0            | 999,999      |
| Peak Queue Size          | 1,000,000    | 1,000        |
| Elapsed Time (s)         | 20.42        | 12.97        |
| Throughput (ticks/s)     | 48,980       | 77,076       |
| Peak Memory (MB)         | 148.67       | 0.09         |

**Interpretation:**
- Objects Created drops from 1,000,000 to ~1,000 -- the pool size --
  because after the queue fills to capacity and backpressure kicks in,
  every subsequent tick reuses an existing object rather than
  allocating a new one.
- Peak Queue Size for the naive run reaches the full tick count,
  because its queue is unbounded and the producer never has to wait
  for the consumer -- this is itself useful diagnostic evidence,
  distinct from the optimized run's queue, which is bounded at
  `pool_size` by design (see `docs/06_object_pool.md`).
- Peak memory drops by roughly three orders of magnitude, because
  we're no longer holding a million transient objects awaiting
  garbage collection at once.
- Throughput improves by ~1.57x in this run, consistent with the
  reduced allocation overhead identified in Chapter 5's profiling --
  though, as always, treat the memory reduction as the primary,
  more hardware-independent signal, and throughput as a secondary,
  hardware-sensitive one.

## Pool Sizing: An Empirical Question, Not a Guess

In the optimized pipeline, the queue is intentionally bounded to
`pool_size` (see the design note in `pipeline_optimized.py` and
`docs/06_object_pool.md`), so `Peak Queue Size` for that run will
naturally sit at or near `pool_size` itself. The more useful signal
for choosing `pool_size` in your own workload is watching
`Objects Created` as you vary it: a small, roughly constant number of
overflow allocations during warm-up is expected and healthy; a number
that scales with total tick count means the pool is too small for the
concurrency level of your producer and consumer. Undersizing the pool
doesn't break correctness (see the overflow fallback in Chapter 6) --
it just means more ticks fall back to real allocation, reducing (but
not eliminating) the benefit.

## Engineering Tradeoffs

- Absolute numbers (826,000 ticks/sec, etc.) are illustrative and
  **will vary** across machines, OS schedulers, and Python builds.
  What should transfer across environments is the *relative*
  improvement and the *shape* of the object-creation curve, not the
  exact throughput figure.
- Benchmarks measure the pipeline in isolation. A real system also
  contends with network I/O, other processes, and multiple consumers
  -- Chapter 8 discusses how that changes the picture.

## What We Learned

- A trustworthy benchmark controls every variable except the one
  being tested (pooled vs. non-pooled allocation).
- Relative improvement is more portable across machines than absolute
  numbers.

## Key Takeaways

- Object pooling's benefit shows up most clearly in `Objects Created`
  and `Peak Memory`, with throughput improving as a downstream effect.
- Pool sizing should be informed by observed peak queue depth, not
  guessed.

## Interview Questions

1. Why do we run both pipelines with the same random seed?
2. If throughput improved by only 5% but memory usage dropped by 60%,
   would you still consider the optimization worthwhile? Why?

## Real Production Notes

Production trading firms often run benchmarks like this continuously
in CI against historical replay data, alerting engineers if a code
change regresses throughput or memory beyond a defined threshold --
sometimes called "performance CI" or "benchmark gating."

## Common Beginner Mistakes

- Comparing a single run of each pipeline and treating the result as
  definitive, without accounting for run-to-run variance.
- Quoting absolute benchmark numbers from someone else's machine as if
  they'll hold true on your own hardware.
